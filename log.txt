1. Executive Summary
This project designs, implements, and evaluates a secure, scalable, and context-aware device provisioning protocol for Bluetooth Mesh networks, framed as a communication protocol engineering project for the subject Computer Communications and Networks.

The project is structured around three core CCN pillars: protocol design and specification, network communication simulation, and performance evaluation. A lightweight ML-based anomaly detection layer is integrated as a behavioral defense mechanism, complementing the cryptographic protocol with network-level threat detection.

The solution combines:
    • Protocol design across Bluetooth Mesh network layers
    • Device-specific identity secrets and AES-based authentication
    • Context-bound transcript verification to prevent relay and misbinding attacks
    • Python-based network simulation with packet-level message exchange
    • ML anomaly detection (Isolation Forest) for behavioral threat identification
    • Performance evaluation using CCN metrics — latency, message overhead, scalability


2. Problem Statement
Bluetooth Mesh networks require secure onboarding of new devices before they can participate in the network. Existing provisioning mechanisms face several networking and security limitations:
    • Shared bootstrap keys create single-points-of-failure across the network
    • Manual PIN/OOB provisioning does not scale to large mesh deployments
    • Relay and misbinding attacks exploit the flooding-based mesh communication model
    • Compromise of one device can undermine the broader network trust model

From a CCN perspective, the core problem is: how do we design a communication protocol that allows a new node to join a mesh network securely, authenticating itself through a structured message exchange, without relying on shared secrets or manual intervention, and without being vulnerable to network-layer attacks such as replay and relay?

7. Threat Model & Security Analysis
7.1 Adversary Capabilities
    • Can eavesdrop on all provisioning bearer traffic
    • Can replay previously captured packets
    • Can delay, reorder, or relay messages between parties
    • Can compromise limited individual devices
    • Cannot break AES-128 or invert SHA-256
    • Cannot read K_device from the device (assumed secure storage)

8. ML-Based Behavioral Anomaly Detection
8.1 Motivation
Cryptographic verification detects attacks at the message level — it rejects invalid tokens. However, certain attack patterns are visible at the network behavior level before any cryptographic check occurs. An ML anomaly detector acts as a pre-filter, flagging suspicious sessions based on communication timing and frequency patterns.

This creates a layered defense architecture:
    • Layer 1 — Protocol level: Context binding and nonce verification (cryptographic)
    • Layer 2 — Network level: Packet format validation and transcript verification
    • Layer 3 — Behavioral level: ML anomaly detection on session timing patterns

14. Success Criteria
The project is considered successful if all of the following are demonstrated:
    • The full 5-message handshake completes successfully in simulation with matching session keys on both sides
    • Replay attack is visibly defeated — replayed Response packets are rejected due to nonce mismatch
    • Relay attack is visibly defeated — transcript hash mismatch detected when ProvisionerID is altered
    • Misbinding attack fails — DeviceID field mismatch detected during provisioner verification
    • Session keys derived independently by both parties are identical
    • ML anomaly detector achieves > 85% accuracy distinguishing normal from attack sessions
    • Performance metrics are within acceptable IoT limits (< 500ms latency, reasonable packet sizes)
